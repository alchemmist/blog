В этой статье мы попробуем написать свой собственный шел, как
альтернатива bash или zsh. Статья носит образовательный характер,
поэтому ставит целью не идеальное решение, а изучение языка
C и понимание внутренней работы командных оболочек на примере
реализации минимально необходимого функционала. Так как кода
достаточно много, в статье для удобства чтения будут показываться
только ключевые моменты, а рабочий код целиком вы сможете найти
в [реопзитории](https://github.com/alchemmist/your-own-shell).

## Базовый каркас

Итак, давайте перед там как реализовать ключевую логику обработки
команд, напишем базовый каркас взаимодействия с пользователем.
Будем использовать `getline` из `stdio`, чтобы считывать команды
пользователя. При этом про многострочный ввод (например после
открывающейся ковычки) мы пока думать не будем.

```c
#include <stdbool.h>
#include <stdio.h>
#include <stdlib.h>

int main() {
    char *cmd;
    size_t buf_len = 0;
    ssize_t read;

    while (true) {
        printf("mysh> ");
        read = getline(&cmd, &buf_len, stdin);
        if (read == -1) {
            break;
        }

        int status = process_input(cmd, read);
        if (status == -1) {
            break;
        }
    }

    free(cmd);
    return 0;
}
```

Тут мы в цикле для удобства пользователя выводим промпт. Над ним
можно будет поработать отдельно, добавив возможность
кастомизировать и выводить любую информацию, но пока ограничемся
статичным выводом.

Далее мы получаем строку от пользоателя. Давайте обратимся к `man
getline`, чтобы понять как обработать значение:

```plaintext
RETURN VALUE
     On  success, getline() and getdelim() return the number of char‐
     acters read, including the delimiter character, but not  includ‐
     ing the terminating null byte ('\0').  This value can be used to
     handle embedded null bytes in the line read.

     At  end  of  file, both functions return -1 with the file stream
     end-of-file indicator set.  On error, both functions  return  -1
     with  the  file  stream error indicator set, and errno is set to
     indicate the error.

     If *lineptr was set to NULL before the  call,  then  the  buffer
     should be freed by the user program even on failure.
```

Как мы видим из документации, `getline` возвращает количество
прочитанных символов, в случае успеха, и значение -1 в случае
прерывания ввода `EOF` или какой-то ошибки. В нашем случае, мы
обработаем ошибку ввода, как сигнал завершения работы шела.
А в случае успеха, вызовем функуцию `process_input` для обработки
полученной команды. По завершении работы. не забываем очистить
память выделенную под хранение текущей команды.

Теперь перейдем к рассмотрению функции `process_input`, в которой
и будет реализована ключевая логика шела — обработка команд. Ниже
минимальный каркас функции `process_input`, для того чтобы наш
пример хотя бы запустился:

```c
int process_input(char *cmd, size_t len) {
    if (!cmd) {
        return -1;
    }

    if (len > 0 && cmd[len - 1] == '\n') {
        cmd[len - 1] = '\0';
        len--;
    }

    if (len == 0) {
        return 0;
    }

    printf("your input: %s\n", cmd);
    return 0;
}
```

Теперь этот шел уже можно будет собрать и запустить:

```sh
gcc -Wall -Wextra -Werror -fsanitize=address,undefined -g -o shell.bin main.c
./shell.bin
```

## Принцип обработки команд

Перед тем, как писать код, стоит понять базовую логику работы шела
согласно POSIX. Конечно, шел можно реализовывать как угодно. И такие
шелы существуют и даже достаточно популярны, например,
[nu](https://www.nushell.sh/) или
[fish](https://fishshell.com/). Но, теряя обратную совместимость
с POSIX, шел теряет возможность выполнения всех тех скриптов, что
уже написаны под классические `sh` или `bash`. Поэтому в учебном
примере мы будем опираться на [стандарт
POSIX](https://pubs.opengroup.org/onlinepubs/9699919799/utilities/V3_chap02.html),
но без чрезмерных деталей.

```mermaid
flowchart LR
    A[󰘦 Tokenize] --> B[󰑕 Expansion]
    B --> C[󰜫 FieldSplitting]
    C --> D[󰈞 Globbing]
    D --> E[󰆴 QuoteRemoval]
    E --> F[󰒆 Parse]
    F --> G[󰘧 Resolve]
    G --> H[󰅶 Execute]
```

Разбор командной строки выполняется в несколько этапов. Ниже мы
рассмотрим несколько упрощённую но корректную работу этого
алгоритма:

1. Входная строка разбивается на токены с учётом кавычек
   и экранирования. То есть уже на этом этапе символ обратного
   слэша может «склеить» пробел с соседними байтами и не дать
   ему стать разделителем токенов. Сам обратный слэш при этом
   остаётся в токене и будет убран позднее на этапе удаления
   кавычек. Внутри апострофов байты сохраняются как есть и
   никакие расширения не выполняются. Внутри кавычек сохраняются
   пробелы и большинство байтов, но допускаются расширения.

   Например, строка `echo "a b" 'c d'` даёт три токена: `echo`,
   `a b`, `c d`.

2. Выполняются расширения: подстановка параметров (`$VAR`,
   `${VAR}`, `$?`, `$1` и т.п.), подстановка команд `$(...)`,
   арифметика `$((...))`. Если переменная не задана, подстановка
   даёт пустую строку. Этот шаг применяется к каждому токену
   и может изменить его содержимое, но сам ещё не делит токены.

   Например, ввод `echo "$HOME" $(printf 2) $((1+2))` будет
   расширен сначала перменной `HOME`, затем символом `2`, который
   получится в результате исполнения `printf` и затем результатом
   арифметической операции `+`. В итоге получистя `echo /home/user 2 3`

3. Выполняется разделение на поля (field splitting) по `IFS` для тех
   фрагментов, которые не находятся в кавычках. `IFS`
   (Internal Field Separator) — это строка с разделителями, обычно
   пробел, таб и перевод строки. Смысл этапа в том, что результат
   предыдущих расширений может содержать пробелы, и их нужно
   превратить в отдельные аргументы команды. То есть shell берёт
   подставленную строку и делит её на слова по символам из `IFS`.
   Если `IFS` содержит несколько разделителей подряд, они считаются
   как один разделитель, и пустые поля обычно не создаются.
   Внутри кавычек этот этап отключён, чтобы пробелы
   сохранились частью аргумента. Этот шаг может превратить один
   токен в несколько токенов.

   Например, в `bash` команда `v="a b"; printf '[%s]\n' $v`
   печатает две строки `[a]` и `[b]`. А `v="a b"; printf '[%s]\n' "$v"`
   печатает одну строку `[a b]`. Это различие и есть смысл
   третьего этапа. Примечание: в `zsh` по умолчанию field
   splitting отключён.

4. Выполняется pathname expansion (globbing) для `*`, `?`,
   `[...]`, если соответствующие символы не защищены кавычками.
   Этот шаг тоже может размножить один токен в несколько токенов.

   Например, `echo *.c` подставляет список файлов C в текущем
   каталоге.

5. Удаляются кавычки (quote removal). Из каждого токена удаляются
   парные кавычки, которые служили для группировки и защиты
   пробелов/символов; экранирующие обратные слэши тоже снимаются. При этом
   сами символы внутри кавычек остаются в токене как есть. Этот
   шаг не делит токены и не изменяет их порядок, он лишь очищает
   их от служебных символов.

   Например, `echo "a\"b"` перед выполнением остаётся `a"b`.

6. Из получившихся токенов строится команда: сначала распознаются
   присваивания вида `NAME=VALUE` и перенаправления (`>`, `<`,
   `2>`, `>>` и т.п.). Первое слово, которое не является ни
   присваиванием, ни перенаправлением, считается именем команды,
   остальные слова — её аргументами. Перенаправления можно писать
   в любом месте — они не попадают в список аргументов,
   а сохраняются отдельно. Если имя команды отсутствует, остаются
   только присваивания (они меняют переменные текущего shell)
   и перенаправления (они применяются к текущему shell).

   Например, `A=1 ls -l > out.txt`: `A=1` это присваивание, `ls`
   имя, `-l` аргумент, `> out.txt` редирект.

7. Выбор, что именно выполнять: special built-in (например,
   `exit`, `cd`, `export`), затем функция, затем обычная builtin,
   и только потом внешний файл через `PATH`.

   Например, при наличии функции `cd()` всё равно выполняется
   встроенная `cd`, потому что это special built-in.

8. Применяются перенаправления и выполняется команда. Сначала shell
   подготавливает файловые дескрипторы: открывает файлы для `>`, `<`,
   `>>`, `2>` и других редиректов, привязывает их к нужным дескрипторам,
   а затем запускает команду с уже перенастроенными входом/выходом.
   Если выбран внешний файл, то используется `fork` + `exec`, а если
   это builtin — выполняется в текущем процессе с теми же редиректами.

   Например, `echo hi > out.txt` создаёт или перезаписывает файл и
   пишет в него.
